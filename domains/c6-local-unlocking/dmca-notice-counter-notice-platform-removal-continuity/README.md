## 研究推进骨架

### 最小问题集

- 界定局部阻塞：某一疾病、流程、服务、职业、家庭、社区或专项风险中的具体失败点。
- 说明受影响任务：它阻碍了主体完成什么行动、服务、照护、转换或恢复过程。
- 记录可测改善：等待时间、错误率、成本、可及性、完成率、伤害率或恢复速度。
- 明确升级边界：何时应转交 C2-C5 的上层生命、能力、转化或基础设施域。
- 避免把局部改良包装成根本性解决方案，必须标注适用范围和外部依赖。

### 变量接口

- 输入变量：本域直接处理的对象、资源、技术、环境、制度或状态。
- 中间机制：变量通过什么因果路径改变主体状态、系统状态或外部条件。
- 状态改变：身体、认知、能力、资源、风险暴露、可及性、时间成本或协作条件如何变化。
- 风险 / 成本函数：死亡风险、失能风险、工程风险、尾部风险、机会成本、维护成本或治理成本如何变化。
- 输出指标：有效寿命、健康寿命、有效时间、主观时间、相对时间、行动能力、恢复能力或未来选择权。

### 证据入口

- 官方 / 原始资料：监管文件、数据库、临床登记、标准、论文原文、项目白皮书或一手报告。

- 数据集 / 登记系统：可复核的数据表、代码、实验记录、登记号、版本和采集边界。
- 反例 / 失败案例：负结果、副作用、安全事故、不可复现结果、伦理争议和误用案例。

`dmca-notice-counter-notice-platform-removal-continuity` 研究 DMCA notice-and-takedown、counter-notice、designated agent、平台移除、恢复窗口、重复侵权政策和版权管理信息如何影响主体表达、收入、作品可达和平台资源连续性。

> 核心问题：版权争议在平台环境里会直接变成内容不可见、账号受限、收入中断和声誉损失；程序边界本身就是创造连续性的基础设施。

## 关注对象

- DMCA Section 512、notice-and-takedown、counter-notice、service provider safe harbor 和 designated agent 记录。
- 平台内容移除、恢复、重复侵权政策、版权 strike、monetization 限制和通知质量。
- false notice / false counter-notice、版权管理信息、权利误报和争议升级风险。
- 创作者、研究者、教育者、开源项目、平台商家和社区知识库的内容可达性。

## Human Infra 链路

```text
作品在平台和网络中传播
  -> 版权通知、移除、反通知和恢复流程改变内容可达性
  -> 内容不可见、收入中断、账号风险和声誉损失改变主体资源状态
  -> 程序透明、记录完整和恢复窗口影响创造连续性
  -> 主体表达、作品资产和未来选择权得到更可审查的保护
```

## 非目标

- 不提供 DMCA notice、counter-notice、申诉、投诉、起诉、索赔、平台策略或法律意见。
- 不判断具体作品是否侵权、是否 fair use、是否应下架、是否应恢复或是否构成虚假通知。
- 不提供版权规避、镜像重传、平台审核规避、重复侵权规避、DRM 绕过或规避版权管理信息的方法。
- 不收集平台账号、争议作品、法律通信、身份材料、权利证明或未公开通知资料。
